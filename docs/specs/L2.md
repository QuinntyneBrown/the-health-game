# L2 - Detailed Requirements

## L2-001: Create Goal
**Traces to:** L1-001

An authenticated user must be able to create a goal with a name, optional description, target metric (numeric value and unit), and cadence.

**Acceptance Criteria:**
1. Given an authenticated user is on the goals screen, when they submit a valid goal (name, target, cadence), then the system shall persist the goal owned by their user account and display it in their goal list.
2. Given an authenticated user submits a goal with an empty name, when the form is submitted, then the system shall reject the request with a validation error and shall not persist any record.
3. Given an authenticated user submits a goal with a non-positive target value, when the form is submitted, then the system shall reject the request with a validation error.
4. Given an unauthenticated request, when it attempts to create a goal, then the system shall respond with HTTP 401 and shall not persist any record.

---

## L2-002: View Goals
**Traces to:** L1-001

An authenticated user must be able to view a list of their goals and the detail of any single goal they own.

**Acceptance Criteria:**
1. Given an authenticated user with goals, when they open the goals list, then the system shall display only goals owned by that user.
2. Given an authenticated user, when they request a goal owned by another user, then the system shall respond with HTTP 403 or 404 and shall not disclose the resource.
3. Given an authenticated user opens a goal detail, when the page loads, then the system shall display the goal name, description, target, cadence, current streak, and longest streak.

---

## L2-003: Edit Goal
**Traces to:** L1-001

An authenticated user must be able to edit any of their own goals' name, description, target, and cadence.

**Acceptance Criteria:**
1. Given an authenticated user owns a goal, when they submit valid edits, then the system shall update the goal and reflect the changes on subsequent reads.
2. Given a user attempts to edit a goal they do not own, when the request is made, then the system shall respond with HTTP 403 or 404 and shall not modify the goal.
3. Given a cadence change, when the goal is updated, then the system shall recompute streak windows from the change date forward without altering historical activity entries.

---

## L2-004: Delete Goal
**Traces to:** L1-001

An authenticated user must be able to delete any of their own goals.

**Acceptance Criteria:**
1. Given an authenticated user owns a goal, when they confirm deletion, then the system shall remove the goal and its associated activity entries and rewards from the user's view.
2. Given a user attempts to delete a goal they do not own, when the request is made, then the system shall respond with HTTP 403 or 404 and shall not delete anything.
3. Given a deletion request without confirmation, when submitted via the UI, then the system shall require an explicit confirmation step before performing the delete.

---

## L2-005: Log Activity Against a Goal
**Traces to:** L1-002

An authenticated user must be able to log an activity entry against one of their goals, recording timestamp, quantity, and optional notes.

**Acceptance Criteria:**
1. Given an authenticated user owns a goal, when they submit an activity entry with a quantity and optional note, then the system shall persist the entry timestamped to the server's UTC clock (or a user-supplied timestamp) and associate it with the goal.
2. Given an authenticated user submits an activity entry with a future timestamp beyond the current cadence window, when the request is made, then the system shall reject it with a validation error.
3. Given a user attempts to log activity against a goal they do not own, when the request is made, then the system shall respond with HTTP 403 or 404.
4. Given an activity entry has been logged, when the user views the goal detail, then the entry shall appear in a chronologically ordered activity history.

---

## L2-006: Edit and Delete Activity Entries
**Traces to:** L1-002

An authenticated user must be able to edit or delete their own activity entries.

**Acceptance Criteria:**
1. Given an authenticated user owns an activity entry, when they edit its quantity or note, then the system shall persist the change and recompute streaks.
2. Given an authenticated user owns an activity entry, when they delete it, then the system shall remove it and recompute streaks.
3. Given a user attempts to edit or delete another user's activity entry, when the request is made, then the system shall respond with HTTP 403 or 404.

---

## L2-007: Compute Current and Longest Streak
**Traces to:** L1-003

The system must compute, for each goal, the current streak (consecutive cadence periods up to and including the current period in which the target was met) and the longest streak ever achieved.

**Acceptance Criteria:**
1. Given a daily goal where the target has been met every day for the past 7 days including today, when the streak is computed, then the current streak shall be 7 and the longest streak shall be at least 7.
2. Given the user fails to meet the target in any complete cadence period, when the next period begins, then the current streak shall reset to 0 (or to the count from the next met period onward) and the longest streak shall be preserved.
3. Given a cadence period is still in progress and the target has not yet been met, when the streak is computed mid-period, then the current streak shall include the most recent fully completed met period(s) and shall not penalize the in-progress period.
4. Given two or more goals belonging to the same user, when streaks are computed, then each goal's streak shall be independent of any other goal's streak.

---

## L2-008: Display Streaks in UI
**Traces to:** L1-003, L1-010

The system must display each goal's current and longest streak prominently in the goal list and goal detail views.

**Acceptance Criteria:**
1. Given a goal exists, when the user views the goal list, then the current streak shall be visible on the goal's card without requiring navigation.
2. Given a goal detail view, when the page loads, then both the current streak and the longest streak shall be displayed.
3. Given a viewport <576px, when streaks are displayed, then the layout shall remain readable and shall not require horizontal scrolling.

---

## L2-009: Define Reward
**Traces to:** L1-004

An authenticated user must be able to define a reward consisting of a name, description, and a qualifying condition tied to either a specific goal achievement or a streak milestone.

**Acceptance Criteria:**
1. Given an authenticated user, when they create a reward bound to a goal with a streak threshold (e.g., "7-day streak"), then the system shall persist the reward and link it to the goal.
2. Given a reward without a qualifying condition, when the form is submitted, then the system shall reject the request with a validation error.
3. Given a user attempts to attach a reward to a goal they do not own, when the request is made, then the system shall reject the request.

---

## L2-010: Earn and Surface Rewards
**Traces to:** L1-004

The system must detect when a reward's qualifying condition is met and must surface the earned reward to the user.

**Acceptance Criteria:**
1. Given a reward tied to a 7-day streak on a goal, when the user's current streak reaches 7, then the system shall mark the reward as earned and notify the user in-app.
2. Given a reward has been earned, when the user views their rewards, then earned rewards shall be visually distinguished from pending rewards and shall display the date earned.
3. Given the qualifying streak is later broken, when the streak resets, then previously earned rewards shall remain earned and shall not be revoked.

---

## L2-011: Goal Cadence — Hourly, Daily, Weekly, Monthly
**Traces to:** L1-005

The system must support goal cadences of hourly, daily, weekly, and monthly, with cadence boundaries computed in the user's local time zone.

**Acceptance Criteria:**
1. Given a daily goal, when the user's local clock crosses midnight, then the cadence period shall roll over and the new period's target shall be reset.
2. Given a weekly goal, when the configured week-start (e.g., Monday) is reached, then the cadence period shall roll over.
3. Given a monthly goal, when the calendar month ends, then the cadence period shall roll over on the first day of the next month.
4. Given an hourly goal, when the system clock crosses the top of the hour, then the cadence period shall roll over.

---

## L2-012: Custom Interval Cadence
**Traces to:** L1-005

The system must allow the user to define a custom cadence as "every N hours" or "every N days" (N >= 1).

**Acceptance Criteria:**
1. Given a user creates a goal with a custom cadence of "every 3 days", when the goal is saved, then the system shall persist the cadence and shall compute period boundaries from the goal's start date in 3-day increments.
2. Given a custom cadence with N <= 0, when submitted, then the system shall reject the request with a validation error.

---

## L2-013: User Registration and Sign-In via PKCE / OIDC
**Traces to:** L1-006, L1-008

The system must allow new users to register and authenticate via an OIDC provider using the PKCE authorization code flow. This flow operates alongside the local username + password flow defined in L2-036; both must produce an equivalent authenticated session.

**Acceptance Criteria:**
1. Given an unregistered visitor, when they complete the OIDC PKCE registration flow, then the system shall create a user record and establish an authenticated session.
2. Given a registered user, when they sign in via PKCE, then the system shall validate the authorization code and code verifier and shall issue tokens with appropriate expirations.
3. Given any authentication flow, when tokens are exchanged, then the system shall not log token values, code verifiers, or any other secrets at any log level.
4. Given a user's session token has expired, when they make a request, then the system shall respond with HTTP 401 and the frontend shall route the user to re-authenticate.

---

## L2-014: User Profile and Account Deletion
**Traces to:** L1-006

The system must allow an authenticated user to view and update their profile and to permanently delete their account.

**Acceptance Criteria:**
1. Given an authenticated user, when they update their display name or email (subject to provider constraints), then the system shall persist the change and reflect it in subsequent views.
2. Given an authenticated user requests account deletion, when they confirm the action, then the system shall delete or anonymize the user's personal data and revoke their active sessions.
3. Given a deleted account, when an attempt is made to sign in with its credentials, then the system shall reject the attempt.

---

## L2-015: Role-Based Access Control End-to-End
**Traces to:** L1-007

The system must enforce RBAC at the database (row ownership / tenant scoping), backend (authorization policies on every endpoint), and frontend (UI gating of role-restricted controls).

**Acceptance Criteria:**
1. Given a user without the `Admin` role, when they call any administrative endpoint, then the system shall respond with HTTP 403.
2. Given a user without the `Admin` role, when they view the UI, then administrative controls shall not be rendered (UI gating must not be the only check — the backend shall still enforce).
3. Given any data-bearing endpoint, when accessed, then the system shall verify the requesting user owns or has explicit access to the requested resource and shall return 403 or 404 otherwise.

---

## L2-016: Input Validation and Injection Defense
**Traces to:** L1-008

The system must validate all inputs at trust boundaries and must defend against injection attacks (SQL, command, header, etc.).

**Acceptance Criteria:**
1. Given any API endpoint, when invalid or malformed input is submitted, then the system shall reject the request with HTTP 400 and a structured error response.
2. Given any database access, when user input is used in a query, then the system shall use parameterized queries (via EF Core) — string-concatenated SQL must not be used.
3. Given any output rendered to the browser, when it includes user-supplied content, then the frontend shall escape it to prevent XSS.
4. Given an authenticated mutating request, when CSRF protection applies, then the system shall require and validate an anti-CSRF token (or equivalent SameSite/origin protections).

---

## L2-017: Secrets and Transport Security
**Traces to:** L1-008

The system must enforce HTTPS in all non-local environments, must store secrets outside source control, and must never log secrets.

**Acceptance Criteria:**
1. Given a request to any non-local environment, when made over plain HTTP, then the system shall redirect to HTTPS or reject the request.
2. Given application configuration, when secrets are required, then they shall be loaded from environment variables or a secret store via Microsoft.Extensions.Configuration — never hard-coded.
3. Given any log entry, when produced by the application, then it shall not contain credentials, tokens, code verifiers, or PII beyond what is strictly necessary for diagnostics.

---

## L2-018: API Performance Targets
**Traces to:** L1-009

The backend API must meet defined response-time targets for typical user workflows.

**Acceptance Criteria:**
1. Given a single-user request to read a goal list of up to 100 goals, when measured at the server, then the p95 response time shall be <= 300ms.
2. Given a single-user request to create or update a goal or activity entry, when measured at the server, then the p95 response time shall be <= 500ms.
3. Given 100 concurrent authenticated users performing typical read/write operations, when measured at the server, then error rate shall remain < 1% and the system shall not exceed defined CPU/memory budgets.

---

## L2-019: Frontend Bundle and Load Performance
**Traces to:** L1-009, L1-010

The frontend must load quickly on a typical mobile connection.

**Acceptance Criteria:**
1. Given a cold load of the main application on a simulated 4G connection, when measured, then the Largest Contentful Paint (LCP) shall be <= 2.5s.
2. Given the main application bundle, when built for production, then the initial JS payload shall be code-split such that route-level chunks are lazy-loaded.
3. Given any view, when interacted with, then Total Blocking Time (TBT) shall be <= 200ms on mid-tier mobile hardware.

---

## L2-020: Responsive Design Across Breakpoints
**Traces to:** L1-010

Every UI screen must render correctly and remain fully usable across XS, S, M, L, and XL viewports.

**Acceptance Criteria:**
1. Given a viewport width <576px, when the user views any screen, then content shall reflow to a single-column layout with no horizontal scrolling and all interactive controls shall remain reachable.
2. Given a viewport width >=576px and <768px, when the user views any screen, then layout shall adapt appropriately for small tablets / large phones.
3. Given a viewport width >=768px and <992px, when the user views any screen, then medium-screen layout (e.g., side-by-side panels where appropriate) shall be used.
4. Given a viewport width >=992px and <1200px, when the user views any screen, then large-screen layout shall be used with sufficient white space.
5. Given a viewport width >=1200px, when the user views any screen, then content shall not stretch unboundedly — a max content width shall be enforced.

---

## L2-021: Accessibility (WCAG 2.1 AA)
**Traces to:** L1-011

Every UI component and screen must meet WCAG 2.1 AA.

**Acceptance Criteria:**
1. Given any interactive control, when navigated via keyboard only, then the control shall be reachable via Tab order, shall display a visible focus indicator, and shall be activatable via Enter or Space as appropriate.
2. Given any text content, when measured against its background, then color contrast shall meet WCAG 2.1 AA minimums (4.5:1 for normal text, 3:1 for large text).
3. Given any image, icon-only button, or non-text control, when rendered, then it shall expose an accessible name (e.g., `aria-label` or visually-hidden text).
4. Given any form field, when rendered, then it shall be associated with a programmatic label.

---

## L2-022: Structured Logging via Microsoft.Extensions.Logging
**Traces to:** L1-012, L1-014

The backend must produce structured logs through `Microsoft.Extensions.Logging` and must include correlation identifiers for requests.

**Acceptance Criteria:**
1. Given any incoming HTTP request, when processed, then the system shall attach a correlation ID to the logging scope and include it in every log entry produced during that request.
2. Given any unhandled exception, when caught by the global handler, then the system shall log it at `Error` level with structured properties (route, user id if available, correlation id) and shall not leak the exception stack trace to the client.
3. Given any security-relevant action (sign-in, role change, account deletion), when performed, then the system shall produce an audit log entry at `Information` level or higher.

---

## L2-023: Clean Architecture Layering
**Traces to:** L1-013, L1-014

The backend must be organized into Domain, Application, Infrastructure, and Api layers, with dependencies pointing inward only.

**Acceptance Criteria:**
1. Given the Domain layer, when inspected, then it shall not reference Application, Infrastructure, or Api projects.
2. Given the Application layer, when inspected, then it shall not reference Infrastructure or Api projects directly — it shall depend on abstractions defined within Application or Domain.
3. Given the Infrastructure layer, when inspected, then it may reference Domain and Application but shall not reference Api.
4. Given the Api layer, when inspected, then it shall compose Application and Infrastructure and shall not contain business rules.

---

## L2-024: SOLID Principles
**Traces to:** L1-013

The entire solution (backend and frontend) must follow S.O.L.I.D. principles.

**Acceptance Criteria:**
1. Given any class, when inspected, then it shall have a single, clearly stated responsibility (SRP).
2. Given a behavior that needs to be extended, when implemented, then extension shall be possible without modifying existing stable abstractions (OCP).
3. Given any abstraction, when implemented by a concrete type, then the implementation shall be substitutable without breaking callers (LSP).
4. Given any consumer, when it depends on an interface, then the interface shall expose only the members the consumer needs (ISP).
5. Given any high-level module, when it collaborates with a low-level module, then it shall depend on an abstraction injected via DI rather than a concrete type (DIP).

---

## L2-025: Backend — MediatR CQS
**Traces to:** L1-014

All backend application use cases must be implemented as MediatR (free version) commands or queries, dispatched from controllers.

**Acceptance Criteria:**
1. Given an HTTP endpoint, when it handles a request, then the controller action shall translate the request into a MediatR `IRequest` (command or query) and dispatch it via `IMediator.Send`.
2. Given a command (state-changing use case), when implemented, then it shall have a corresponding `IRequestHandler` and shall return either `Unit` or a result DTO.
3. Given a query (read use case), when implemented, then it shall have a corresponding `IRequestHandler` and shall return a result DTO; queries shall not mutate state.
4. Given the MediatR package, when added to the project, then it shall be the free OSS version (not the paid commercial license tier).

---

## L2-026: Backend — ASP.NET Core Controllers
**Traces to:** L1-014

The backend HTTP surface must be implemented using ASP.NET Core Controllers (not Minimal APIs).

**Acceptance Criteria:**
1. Given any HTTP endpoint, when implemented, then it shall be defined on a class that inherits from `ControllerBase` (or `Controller`) and decorated with `[ApiController]`.
2. Given any controller action, when invoked, then it shall return an `IActionResult` (or typed `ActionResult<T>`) and shall delegate work to MediatR.

---

## L2-027: Backend — Entity Framework DbContext
**Traces to:** L1-014

The backend must use Entity Framework Core with a `DbContext` for data access against Microsoft SQL Server.

**Acceptance Criteria:**
1. Given any data access, when performed, then it shall go through an EF Core `DbContext` — direct ADO.NET or Dapper usage is not permitted for primary persistence.
2. Given the database engine, when the application connects, then it shall connect to Microsoft SQL Server.
3. Given schema changes, when introduced, then they shall be expressed as EF Core migrations checked into source control.

---

## L2-028: Backend — Microsoft Extensions for Logging, Configuration, DI
**Traces to:** L1-014

The backend must use `Microsoft.Extensions.Logging`, `Microsoft.Extensions.Configuration`, and `Microsoft.Extensions.DependencyInjection` (the built-in ASP.NET Core providers).

**Acceptance Criteria:**
1. Given any service registration, when performed, then it shall be registered with the built-in `IServiceCollection` DI container.
2. Given any configuration value, when read, then it shall be obtained via `IConfiguration` or `IOptions<T>`.
3. Given any logging output, when produced, then it shall be written via `ILogger<T>` injected from `Microsoft.Extensions.Logging`.

---

## L2-029: Backend — Solution Structure
**Traces to:** L1-015

The backend must use the old-style `.sln` solution format and must be organized into `backend/src` and `backend/tests`.

**Acceptance Criteria:**
1. Given the repository, when inspected, then a `backend/` folder shall exist containing a `.sln` file in the legacy (Visual Studio old-style) format.
2. Given the solution, when opened, then production projects shall live under `backend/src/` (including the Api project) and test projects shall live under `backend/tests/`.

## L2-030: Frontend — Angular Workspace and Material
**Traces to:** L1-016

The frontend must be a single Angular workspace, must use Angular Material for every component, and must apply Design Tokens.

**Acceptance Criteria:**
1. Given the frontend repository, when inspected, then it shall contain a single Angular workspace (`angular.json`) with multiple libraries and at least one application.
2. Given any UI element (button, header, input, etc.), when rendered, then it shall be implemented using an Angular Material component or a wrapper around one — raw native elements styled from scratch shall not be used for these UI primitives.
3. Given any color or spacing value used in a component or library, when authored, then it shall reference a Design Token (CSS custom property or token-mapped Sass variable) — hard-coded hex values or pixel values for color/spacing shall not appear in component styles.

---

## L2-031: Frontend — BEM and File-Per-Type
**Traces to:** L1-017

The frontend must enforce BEM naming for HTML/CSS classes and must split components into separate files per type.

**Acceptance Criteria:**
1. Given any component, when authored, then it shall consist of separate `.ts`, `.html`, and `.scss` files (single-file components with inline templates or styles are not permitted).
2. Given any HTML/CSS class, when authored, then it shall follow BEM naming (`block`, `block__element`, `block--modifier`).

---

## L2-032: Frontend — Interface-Driven Service Consumption
**Traces to:** L1-017

Every Angular service that is consumed by another component or service must expose a TypeScript interface and an `InjectionToken`, per https://github.com/QuinntyneBrown/interface-driven-service-consumption.

**Acceptance Criteria:**
1. Given a service `FooService`, when authored, then there shall be a sibling file `foo.service.contract.ts` that exports a TypeScript interface (e.g., `IFooService`) and an `InjectionToken<IFooService>`.
2. Given a consumer of the service, when injecting it, then the consumer shall inject via the `InjectionToken` and reference only the interface — it shall not import the concrete `FooService` class.
3. Given the providers configuration, when set up, then the `InjectionToken` shall be bound to the concrete implementation (`useExisting` or `useClass`).

---

## L2-033: Frontend — Library Layout and Dependencies
**Traces to:** L1-018

The Angular workspace must contain `api`, `components`, and `domain` libraries plus a main application, with the dependency rules below enforced.

**Acceptance Criteria:**
1. Given the `api` library, when inspected, then it shall contain only models and backend-facing services (with their contracts), and it shall not depend on `components` or `domain`.
2. Given the `components` library, when inspected, then it shall contain reusable UI components and shall not depend on `api` or `domain`.
3. Given the `domain` library, when inspected, then it shall depend on `api` and may depend on `components`.
4. Given the main application, when inspected, then it shall depend on `api`, `components`, and `domain`.
5. Given any forbidden dependency (e.g., `components` -> `api`), when introduced, then the build (or a lint rule such as Nx module boundaries / `eslint-plugin-import`) shall fail.

## L2-034: Unit and Integration Testing
**Traces to:** L1-019

The solution must include automated unit tests for application logic and integration tests for backend command/query handlers.

**Acceptance Criteria:**
1. Given any MediatR handler, when added or modified, then it shall be covered by at least one unit or integration test.
2. Given any non-trivial domain logic (e.g., streak computation), when implemented, then it shall be covered by unit tests including boundary cases (period rollover, missed period, partial period).

## L2-035: End-to-End Testing with Playwright POM
**Traces to:** L1-019

End-to-end tests must use Playwright and must follow the Page Object Model pattern, covering important user-facing functionality.

**Acceptance Criteria:**
1. Given an E2E test suite, when authored, then page interactions shall be encapsulated in Page Object classes — tests shall not call `page.locator(...)` directly inside test bodies.

3. Given any acceptance test (unit, integration, or E2E), when authored, then the test file shall include a comment header in the form `// Acceptance Test\n// Traces to: L2-NNN[, L2-NNN...]\n// Description: ...`.

## L2-036: Username + Password Sign-In with JWT Access Tokens
**Traces to:** L1-006, L1-008

The system must support local sign-in with a username (or email) and password. The backend validates credentials against user records persisted in the application database and, on success, issues a signed JWT access token; this flow must coexist with the PKCE / OIDC flow defined in L2-013 and must produce an equivalent authenticated session.

**Acceptance Criteria:**
1. Given the frontend, when the user navigates to the sign-in route, then a sign-in page shall be displayed with username (or email) and password fields and a submit control.
2. Given valid credentials are submitted, when the backend authenticates the user, then it shall verify the password against the stored password hash, issue a signed JWT access token (and a refresh token where supported), and respond with HTTP 200 and the token payload.
3. Given invalid credentials are submitted, when the backend processes the request, then it shall respond with HTTP 401 and a generic error message that does not disclose whether the username or the password was incorrect, and shall not establish a session.
4. Given a user record exists, when the password is persisted, then it shall be stored only as a salted hash produced by a modern password-hashing function (e.g., Argon2id, PBKDF2 with adequate iterations, or bcrypt with adequate cost) — plaintext passwords shall never be stored or logged.
5. Given a successful sign-in, when the access token is issued, then it shall be a JWT signed with a key sourced from configuration / a secret store, shall include the user's identifier and role claims, and shall have a defined expiration; the token shall be transmitted over HTTPS and shall not be logged at any level.
6. Given a JWT access token presented on a subsequent request, when the backend processes it, then it shall validate the issuer, audience, signature, and expiration before honoring the request; an invalid or expired token shall produce HTTP 401.
7. Given repeated failed sign-in attempts for the same account, when a configured threshold is exceeded, then the system shall apply a rate-limit, lockout, or equivalent throttle and shall log the security event at `Information` level or higher (without logging the submitted password).
8. Given a sign-in attempt for a deleted or disabled account, when processed, then the backend shall reject it with HTTP 401 regardless of credential correctness.
