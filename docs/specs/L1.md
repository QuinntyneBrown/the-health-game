# L1 - High-Level Requirements

## L1-001: Goal Definition
The system must allow an authenticated user to create, view, edit, and delete personal health goals. Each goal has a name, description, target metric, and a cadence (hourly, daily, weekly, monthly, or a custom user-defined interval).

## L1-002: Activity Logging
The system must allow an authenticated user to log activity entries against their goals, including the timestamp, quantity (where applicable), and optional notes.

## L1-003: Streak Tracking
The system must compute and display the user's current and longest streak for each goal, where a "streak" is an unbroken run of consecutive cadence periods in which the goal target was met.

## L1-004: Reward Definition & Earning
The system must allow a user to define rewards tied to specific goals or streak milestones, and must surface earned rewards to the user when the qualifying condition is met.

## L1-005: Goal Cadence and Custom Intervals
The system must support goal cadences of hourly, daily, weekly, and monthly, and must allow users to define custom intervals (e.g., every N hours / N days). The cadence drives streak computation and target evaluation windows.

## L1-006: User Management and Authentication
The system must provide full user management — registration, sign-in, sign-out, password reset, profile management, and account deletion. The backend must store user records in the application database, validate user credentials, and implement OAuth2-compatible token issuance for successful sign-ins. A sign-in page collects a username or email and password, the backend validates those credentials against stored password hashes, and the backend issues a signed JWT access token (with a refresh token where appropriate).

## L1-007: Authorization and Role-Based Access Control
The system must enforce Role-Based Access Control (RBAC) end-to-end, from database row ownership through backend authorization checks to frontend UI gating, so that users can only access and modify their own data and administrative actions are restricted to authorized roles.

## L1-008: Security and Data Protection
The system must protect against the OWASP Top 10 (injection, broken access control, sensitive data exposure, XSS, CSRF, SSRF, insecure deserialization, etc.), enforce HTTPS in transit, validate all inputs at trust boundaries, and store secrets and credentials securely.

## L1-009: Performance and Scalability
The system must meet defined response-time and throughput targets for typical user workflows and must remain responsive under expected concurrent-user load.

## L1-010: Responsive, Mobile-First UI
The system's web frontend must be designed mobile-first and must render correctly and remain fully usable across XS (<576px), S (>=576px), M (>=768px), L (>=992px), and XL (>=1200px) viewport breakpoints.

## L1-011: Accessibility
The system's UI must conform to WCAG 2.1 AA accessibility guidelines, including keyboard navigation, screen-reader compatibility, sufficient color contrast, and focus indicators.

## L1-012: Observability and Logging
The system must produce structured, queryable logs for application events, errors, and security-relevant actions, using Microsoft.Extensions.Logging on the backend.

## L1-013: Solution Architecture and SOLID
The entire solution (backend and frontend) must follow S.O.L.I.D. principles, and the backend must be organized using Clean Architecture (with clear separation of Domain, Application, Infrastructure, and API layers).

## L1-014: Backend Technology Stack
The backend must be implemented in .NET, using ASP.NET Core Controllers for HTTP endpoints, MediatR (free version) for the Command/Query Separation pattern, Entity Framework Core with a `DbContext` for data access against Microsoft SQL Server, and Microsoft Extensions for Logging, Configuration, and Dependency Injection.

## L1-015: Backend Solution Structure
The backend must use the old-style `.sln` solution file format and must be organized into `backend/src` (production projects, including the Api) and `backend/tests` (test projects).

## L1-016: Frontend Technology Stack
The frontend must be implemented in Angular as a single workspace split into libraries and applications, must use Angular Material components for every UI element (buttons, headers, inputs, etc.), and must apply Design Tokens for consistent colors and spacing across the app.

## L1-017: Frontend Code and Component Conventions
The frontend must enforce BEM naming for all HTML/CSS classes, must split every component into separate files per type (`.html`, `.scss`, `.ts`) with no single-file components, and must use interface-driven service consumption (per https://github.com/QuinntyneBrown/interface-driven-service-consumption) — every service exposes a TypeScript interface and an injection token in a `*.service.contract.ts` file.

## L1-018: Frontend Library Structure
The frontend Angular workspace must be organized into an `api` library (models and backend-facing services), a `components` library (reusable UI components with no dependency on `api`), a `domain` library (UI components that depend on `api`), and a main application that depends on `api`, `components`, and `domain`.

## L1-019: Testing Strategy
The system must include automated tests at unit, integration, and end-to-end levels. End-to-end tests must use Playwright with the Page Object Model pattern and must cover important user-facing functionality. Acceptance tests must trace back to L2 requirements via comment headers.
